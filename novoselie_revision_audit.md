# 🔍 Ревизия переезда: что осталось на ноутбуке

> **Кому.** Тем, кто уже прошёл «Новоселье» и заметил: агент на сервере знает не всё, что знал на ноутбуке. Забывает правила, не видит наставников, не помнит договорённости.
>
> **Что это.** Не новый переезд. Проверка фактами + аккуратный доперенос того, что не доехало.
>
> **Как запускать.** Открой Claude Code в папке своего проекта и дай ему ссылку на этот файл (или вставь его целиком). Дальше он работает сам: проверяет фактами, достраивает недостающее, синхронизирует бота. Тебя дёрнет только на трёх вещах — если в перенос попало личное, если надо трогать конфиг бота, если что-то необратимое.
>
> **Сколько.** 20–30 минут, из них твоего внимания — пара минут. Необратимого нет: сначала бэкап, потом репетиция, только потом перенос. Ничего не удаляется.

---

## Почему это вообще случилось (прочитай, чтобы понимать, что проверяем)

Твой «второй мозг» — это не одна папка. Это **шесть разных мест**, и пять из них лежат в скрытых системных путях, а не в папке проекта:

| Что | Где лежит на самом деле | Едет ли при обычном переезде |
|---|---|---|
| **Память** (заметки, контекст, наработки) | `~/.claude/projects/<путь-проекта-через-дефисы>/memory/` | ❌ вне папки проекта — часто не замечают |
| **Скиллы** (твоя команда, наставники) | `~/.claude/skills/` | ⚠️ переносят, но кладут не туда, где агент их ищет |
| **CLAUDE.md** (правила работы агента) | корень папки проекта + `~/.claude/CLAUDE.md` | ❌ почти всегда забывают |
| **Настройки и хуки** (авто-маршрутизация, разрешения) | `.claude/settings.json`, `.claude/hooks/` | ❌ не переносят |
| **Субагенты и команды** | `.claude/agents/`, `.claude/commands/` | ❌ не переносят |
| **MCP-подключения** (календарь, браузер, диск) | `~/.claude.json` | ❌ не переносят |

И седьмое место — **твой Telegram-бот**. Он живёт на сервере отдельно и может читать не ту папку памяти, в которую переехал мозг. Снаружи это выглядит как «бот поглупел»: он жив, отвечает, но из старого слепка.

Отсюда главный эффект: **файлы физически на сервере, но лежат по путям, куда агент не заглядывает.** Он не «забыл» — он туда не смотрит. Память подхватывается автоматически только из `~/.claude/projects/<слуг-проекта>/memory/`, скиллы — только из `~/.claude/skills/`. Всё остальное агент найдёт, лишь если ему прямо сказать «читай вон там».

Второй эффект — **замерзание**. Переезд обычно делают копией «один раз». Дальше ты работаешь на ноутбуке, а серверная копия остаётся в дне переезда. Через две недели агент на сервере живёт в прошлом.

Ревизия закрывает оба.

## ИНСТРУКЦИЯ ДЛЯ АГЕНТА

Ты проводишь ревизию переезда второго мозга на сервер. Правила жёсткие:

- **Только факты с диска.** Каждое утверждение — результат выполненной команды. Формулировки «должно быть перенесено», «скорее всего есть» запрещены. Не знаешь — выполни команду и посмотри.
- **Работай до результата, не дёргай человека по мелочам.** Всё безопасное делай сам: инвентаризация, бэкапы, репетиция, доперенос (он только добавляет файлы), подключение путей, ежечасная синхронизация. Отчитывайся по ходу, но не жди подтверждения на каждый шаг.
- **Останавливайся и спрашивай ровно в трёх случаях:**
  1. в список переноса попало **личное** (здоровье, семья, финансы, секреты) — покажи и жди решения;
  2. нужно **править конфиг бота** или перезапускать сервис;
  3. что-то **необратимое**: удаление, перезапись более новых файлов, смена доступов.

  Во всех остальных случаях — действуй.
- **Ничего не удаляем.** Только `--update` (побеждает более новый файл), никогда `--delete`. Бэкап до, план отката после.
- **Сначала определи ОС пользователя** (`uname -s` или PowerShell) и дальше говори на её языке. Команды ниже даны в Mac-варианте; если у пользователя Windows — переводи их в PowerShell сам (`Get-ChildItem`, `Measure-Object`, `robocopy /XO`, встроенный `scp`). Не заставляй человека подбирать эквиваленты.
- **Личный контур не едет.** Здоровье, семья, финансы, секреты — остаются на локальной машине. Перед переносом покажи список того, что поедет.
- **Секреты не в чат.** IP/логин/пароль берёшь из Файла доступа (macOS: `~/.config/brain/server_access`, Windows: `%USERPROFILE%\.secrets\brain\server_access.txt`). В чат не выводишь никогда.
- **Машина пользователя — Mac или Windows.** Linux в этой инструкции встречается только там, где речь о самом сервере: он на Ubuntu. Не предлагай пользователю линуксовых команд для его компьютера.

### ШАГ 1 — Инвентаризация локальной машины (ничего не меняем)

Найди и посчитай **фактическими командами**, а не по памяти:

1. **Папка памяти.** Определи слуг проекта: абсолютный путь папки проекта, где все `/` заменены на `-`. Пример: `/Users/ivan/Documents/brain` → `-Users-ivan-Documents-brain`. Память лежит в `~/.claude/projects/<слуг>/memory/`.
   ```
   ls ~/.claude/projects/            # покажи ВСЕ проекты — их может быть несколько
   find ~/.claude/projects/*/memory -type f -name "*.md" | wc -l
   ```
   ⚠️ Если проектов с непустой `memory/` несколько — покажи их все пользователю. Частая причина «агент забыл»: контекст разложен по разным папкам, потому что Claude Code запускали из разных мест.
2. **Скиллы:** `ls ~/.claude/skills | wc -l` и `ls <папка-проекта>/.claude/skills 2>/dev/null | wc -l`
3. **CLAUDE.md:** есть ли `<папка-проекта>/CLAUDE.md` и `~/.claude/CLAUDE.md`, размер каждого в килобайтах
4. **Настройки и хуки:** `ls <папка-проекта>/.claude/settings*.json <папка-проекта>/.claude/hooks/ 2>/dev/null`
5. **Субагенты и команды:** `ls ~/.claude/agents ~/.claude/commands <папка-проекта>/.claude/agents <папка-проекта>/.claude/commands 2>/dev/null`
6. **MCP:** есть ли `~/.claude.json` и `<папка-проекта>/.mcp.json`, какие серверы подключены (имена, БЕЗ ключей)

Выведи **таблицей**: что / путь / сколько файлов.

### ШАГ 2 — Инвентаризация сервера теми же мерками

Подключись по SSH (данные из Файла доступа) и посчитай ровно то же самое **на сервере**:

```
ls ~/.claude/skills | wc -l                       # 0 — значит команда не заехала
ls ~/.claude/projects/                            # есть ли проект мозга
find ~/memory -type f -name "*.md" | wc -l        # если память клали в ~/memory
ls -la ~/CLAUDE.md                                # есть ли правила
ls ~/.claude/agents ~/.claude/commands 2>/dev/null
```

Проверь дополнительно, **где ещё** на сервере могли осесть файлы: `/opt/brain`, `/home/*/memory`, `/home/*/skills`, домашняя папка. Часто память лежит в `~/memory`, а агент ищет в `~/.claude/projects/.../memory` — это и есть разрыв.

**Обязательно проверь свежесть, а не только количество:**
```
find ~/memory -type f -mtime -7 | wc -l           # сколько файлов моложе недели
```
Если ответ 0 или 1 — сервер **заморожен**: копия сделана один раз и с тех пор не обновлялась. Назови дату последнего изменения явно.

### ШАГ 3 — Дельта и вердикт

Собери два списка файлов (локально и на сервере) и покажи разницу **числом и именами**:

```
# локально
cd ~/.claude/projects/<слуг>/memory && find . -type f -name "*.md" | sed 's|^\./||' | sort > /tmp/local.txt
# на сервере
cd ~/memory && find . -type f -name "*.md" | sed 's|^\./||' | sort > /tmp/server.txt
# что не доехало
comm -23 /tmp/local.txt /tmp/server.txt
# что есть только на сервере (это НЕЛЬЗЯ потерять — заберём на ноутбук)
comm -13 /tmp/local.txt /tmp/server.txt
```

Выведи вердикт таблицей — по каждому из шести контуров: **✅ на месте / ⚠️ лежит не там, где агент ищет / ❌ не доехало**. И отдельной строкой: сколько дней сервер отстаёт.

Покажи таблицу и **сразу иди дальше** — человеку не нужно ничего подтверждать, чтобы ты сделал бэкап и перенёс недостающее. Останавливайся только по трём правилам из начала файла.

> **Если переезда вообще не было** (на сервере пусто, Claude Code не установлен) — не начинай ревизию, скажи об этом прямо и отправь человека к модулю `02_ЯДРО_ПЕРЕЕЗД.md`: ревизовать нечего, нужно сначала переехать.
>
> **Если сервер недоступен** (SSH не отвечает) — остановись на этом, покажи ошибку и не выдумывай состояние сервера.

### ШАГ 4 — Доперенос

Порядок строгий, выполняешь сам:

1. **Бэкап обеих сторон.** На сервере: `tar czf ~/brain_backup_$(date +%Y%m%d_%H%M%S).tar.gz ~/memory ~/.claude/skills`. Локально — архив папки памяти. Покажи размеры.
2. **Сначала забери с сервера то, чего нет локально** (из ШАГа 3). Ничего не теряем.
3. **Репетиция:** те же команды переноса с `--dry-run`. **Проверь список сам:** если в нём нет ничего личного — иди дальше без вопросов. Если личное мелькнуло — стоп, покажи человеку, разберись с исключениями.
4. **Перенос** — без `--dry-run`, с исключениями личного контура:
   ```
   rsync -az --update --exclude 'personal/' --exclude 'personal/**' \
     --exclude '.secrets' --exclude '*.session' \
     <локальная-память>/ <сервер>:<серверная-память>/
   rsync -az --update <локальные-скиллы>/ <сервер>:~/.claude/skills/
   ```
5. **Подключи то, что агент ищет автоматически** (главное место, где ломается):
   - скиллы — именно в `~/.claude/skills/` на сервере, не в `~/skills`;
   - `CLAUDE.md` — в корень рабочей папки на сервере, и в нём **явными строками** укажи, где лежит память и скиллы;
   - если память лежит не в `~/.claude/projects/<слуг>/memory/` — либо перенеси туда, либо пропиши путь в `CLAUDE.md` первой строкой, чтобы агент читал индекс памяти на старте.
6. **Хуки и настройки** переноси только после того, как покажешь пользователю их содержимое: в них бывают пути локальной машины, которые на сервере не существуют. Пути поправь под сервер.
7. **MCP** — подключения, завязанные на локальные программы (браузер, IDE), на сервере не заработают. Скажи об этом прямо, не переноси молча.

### ШАГ 5 — Проверка фактом (без неё работа не считается сделанной)

Прогони и покажи вывод:

```
ls ~/.claude/skills | wc -l                  # совпадает с локальным числом?
find <серверная-память> -name "*.md" | wc -l # совпадает?
ls -la ~/CLAUDE.md                           # существует?
comm -23 /tmp/local.txt /tmp/server.txt      # пусто?
```

И финальный живой тест: **запусти агента на сервере и спроси его о договорённости, которая была записана после дня переезда.** Ответил по существу — контур подключён. Ответил «не нахожу» — файлы лежат не там, где он ищет, вернись к пункту 5 ШАГа 4.

✅ **Получилось, если**: числа совпали, список «не доехало» пуст, `CLAUDE.md` на месте, и агент на сервере отвечает про недавнюю договорённость.

### ШАГ 5.5 — Бот: чтобы он говорил из свежей памяти

Бот на сервере — отдельный житель, и он **может быть жив, отвечать и при этом читать старую папку памяти**. При переезде это случается чаще всего: мозг переехал в одно место, а бот по-прежнему смотрит в то, откуда его когда-то запускали. Снаружи выглядит как «бот поглупел».

Проверь фактами:

1. **Жив ли бот и как называется его сервис:**
   ```bash
   systemctl list-units --type=service | grep -i bot
   systemctl is-active <сервис>
   ```
2. **Какую память он на самом деле читает:**
   ```bash
   systemctl cat <сервис> | grep -E "WorkingDirectory|EnvironmentFile|ExecStart"
   grep -rn "memory" <папка-бота>/*.py | head
   ```
3. **Сведи пути.** Сравни найденный путь с тем, куда мы только что перенесли память. Совпадает — хорошо. Не совпадает — это и есть причина, по которой бот отвечает старьём: покажи мне оба пути и предложи, как свести (обычно правится одна строка в конфиге бота).
4. **Поисковый индекс.** Если у бота есть RAG/эмбеддинги — переиндексируй после обновления памяти. Иначе он ищет по старому слепку, даже когда файлы уже новые.
5. **Перезапусти и посмотри журнал:**
   ```bash
   systemctl restart <сервис>
   journalctl -u <сервис> -n 30 --no-pager
   ```
   Ошибок быть не должно.

✅ **Живой тест:** напиши боту в Telegram вопрос про заметку, которая появилась у тебя **на днях**. Ответил по существу — контур сведён. Ответил «не знаю» — бот читает не ту память, возвращайся к пункту 3.

### ШАГ 6 — Чтобы сервер больше не замёрз (обязательно)

Разовая копия протухает за неделю. Поставь регулярный перенос — **ежечасно**, средствами своей ОС:

- **macOS:** LaunchAgent (`~/Library/LaunchAgents/`, `StartInterval 3600`) со скриптом переноса.
- **Windows:** Планировщик заданий, задача каждый час, скрипт PowerShell (`robocopy /XO` + встроенный `scp`).

Никаких сторонних программ и подписок — всё встроено в твою систему.

В скрипте — те же исключения личного контура и обязательно `--update` без `--delete`.

Проверка фактом: подожди час (или запусти задачу вручную), измени один файл локально, убедись, что он появился на сервере. Затем поставь себе напоминание через неделю: `find <серверная-память> -mtime -2 | wc -l` — если ноль, синк умер, чини.

## Что сказать агенту, если он ошибается

- Отвечает «всё перенесено» без вывода команды → «Покажи вывод команды с диска. Ответ по памяти не принимаю».
- Просит пароль в чат → «Возьми из Файла доступа».
- Предлагает `rsync --delete` → «Нет. Только `--update`, ничего не удаляем».
- Хочет тащить личное на сервер → «Личный контур остаётся здесь. Покажи список того, что поедет, до переноса».
